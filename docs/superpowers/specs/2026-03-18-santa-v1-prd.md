# Santa v1 PRD

## Product Overview

**Product:** Santa — a package manager meta-tool

**One-liner:** Declare your packages once, install them everywhere.

**Problem:** Developers work across macOS and Linux machines with different package managers. Keeping a consistent set of tools installed means maintaining separate brew/apt/pacman/etc. commands, remembering platform-specific package names, and hoping nothing breaks. There's no single source of truth for "what I want installed."

**Solution:** Santa lets you declare packages in a single CCL config file. It resolves cross-platform package names, generates safe install scripts for each platform's native package manager, and reports what's installed vs. missing. You review and run the scripts — santa never executes anything without your explicit consent (unless you opt in with `-x`).

**Target audience:** Developers who use multiple machines or platforms and want a declarative, safe way to manage their toolchain.

**v1 semver contract:** `santa-cli` and `santa-data` crates at 1.0.0. Public CLI interface and config format are stable — breaking changes only in 2.0. Sickle crates remain at 0.x with no stability guarantee.

## v1 Scope — Commands & Features

### Stable CLI Surface

| Command | Description |
|---|---|
| `santa status` | Show package status with all/installed/missing filters |
| `santa install` | Generate install scripts (or execute with `-x`) |
| `santa add` | Add packages to user config |
| `santa remove` | Remove packages from config (optional uninstall) |
| `santa config` | Show current config or package list |
| `santa completions` | Generate shell completions (bash, zsh, fish, powershell) |
| `santa sources update` | Download latest source definitions |
| `santa sources list` | List available sources (with origin filter) |
| `santa sources show` | Show details about a specific source |
| `santa sources clear` | Revert to bundled sources |

### New Command for v1

| Command | Description |
|---|---|
| `santa init` | Interactive config generation — detects platform, asks what managers to use, creates starter `config.ccl` |

### Global Flags

- `-x, --execute` — Direct execution mode (safe script generation is default)
- `-b, --builtin-only` — Load only default config
- `-v, --verbose` — Increase logging verbosity
- `--format` — Script format (auto/shell/powershell)
- `--output-dir` — Output directory for generated scripts
- `--config` — Custom config file path

### Execution Model

Safe-by-default: all commands generate scripts for user review. The `-x` flag opts into direct execution. This is a core design principle, not a temporary limitation.

### Not in v1 (explicitly deferred)

- `santa search` — cross-manager package search
- Dev-tools pipeline binaries — internal tooling, not public API
- Plugin system for community package managers
- Machine sync / lockfiles
- Windows support (best-effort, not guaranteed)

## Platform & Package Manager Support

### Required Platforms

| Platform | Support Level |
|---|---|
| macOS | Full support, required |
| Linux | Full support, required |
| Windows | Best-effort, not guaranteed |

### Package Manager Tiers

**Tier 1 — Required for v1 (must work reliably):**

| Manager | Platform | Notes |
|---|---|---|
| brew | macOS (+ Linux homebrew) | Primary macOS manager |
| apt | Linux (Debian/Ubuntu) | Primary Linux manager |
| pacman | Linux (Arch) | Required Arch support |
| cargo | Cross-platform | Rust ecosystem |
| npm | Cross-platform | Node.js ecosystem |

**Tier 2 — Shipped but best-effort:**

| Manager | Platform | Notes |
|---|---|---|
| dnf | Linux (Fedora/RHEL) | Fedora ecosystem |
| nix | Cross-platform | Nix ecosystem |
| aur | Linux (Arch) | AUR helper packages |
| arch | Linux (Arch) | Official Arch repos |
| scoop | Windows | Windows manager |
| winget | Windows | Microsoft package manager |
| flathub | Linux | Flatpak packages |

Tier 1 managers must have:
- Correct install/check command templates
- Tested cross-platform name mappings for common packages
- Reliable status detection
- Good error messages on failure

Tier 2 managers ship as-is. They work but aren't held to the same bar. Bugs are fixed on a best-effort basis.

## Configuration & Data Model

### Config Format

CCL (Categorical Configuration Language) is the sole config format for v1. The config lives at `~/.config/santa/config.ccl` by default (overridable with `--config` or `SANTA_CONFIG_PATH`).

### Config Structure

A user config declares which packages they want, grouped by manager. CCL uses `key = value` syntax with 2-space indentation for nesting and empty-key `= item` syntax for list items:

```ccl
/= My packages

brew =
  = ripgrep
  = fd
  = jq

apt =
  = ripgrep
  = fd-find
  = jq

cargo =
  = bat
  = eza
```

### Data Resolution Layers

Santa resolves package data through multiple layers (highest priority first):

1. **User config** — the user's `config.ccl`
2. **Custom sources** — user-added source definitions
3. **Downloaded sources** — fetched via `santa sources update`
4. **Bundled sources** — shipped with the binary

### v1 Config Requirements

- **Documentation:** Config format must be fully documented — what's valid, what each section means, examples for common setups
- **`santa init`:** Generates a working starter config by detecting the current platform and available managers
- **Validation:** Config errors must produce clear, actionable messages (file, line, what's wrong, how to fix)
- **YAML migration:** The existing YAML-to-CCL migration path continues to work but is not documented for new users. CCL is the only documented format.

### Source Definitions

Each package manager is defined in a source file (`sources/*.ccl`) containing:
- Install command template
- Check/status command template
- Package name prefix (if any)
- Platform-specific overrides

These are bundled with the binary and updatable via `santa sources update`.

## Quality Bar

### Error Messages

All user-facing errors must be:
- **Actionable** — tell the user what to do, not just what went wrong
- **Contextual** — include file path, line number, and relevant config snippet where applicable
- **Categorized** — distinguish between config errors, missing managers, network failures, and package resolution failures

Priority areas for error message polish:
- Config parsing errors (invalid CCL, unknown managers, bad structure)
- Package not found in a manager
- Manager not installed on the system
- Network errors during `sources update`
- Script generation failures

### Documentation

v1 must ship with:
- **README** — install instructions, quick start, value prop
- **Config guide** — complete CCL config reference with examples
- **CLI reference** — all commands, flags, and usage patterns (already auto-generated via `just markdown-help`)
- **Troubleshooting guide** — common issues and fixes

### Test Coverage

Before v1, the following must have test coverage:

| Area | Current State | v1 Requirement |
|---|---|---|
| Config parsing & validation | Exists | Expand edge cases, bad input |
| All CLI commands | Partial | Every command has at least happy-path + error-path tests |
| Script generation per Tier 1 manager | Exists | All 5 Tier 1 managers tested |
| Cross-platform name resolution | Partial | Tier 1 managers have verified mappings |
| `santa init` | New | Full test coverage |
| Error message quality | Minimal | Snapshot tests for key error scenarios |
| Security (injection prevention) | Exists | Maintain, no regressions |
| Property-based tests | Exists | Maintain, no regressions |

### Distribution

v1 ships via:
- **Pre-built binaries** on GitHub Releases (macOS aarch64 + x86_64, Linux x86_64 + aarch64)
- **`cargo install santa`** on crates.io
- **Homebrew tap** for macOS (and Linux homebrew) users

CI must produce release artifacts for all supported targets.

## Non-Goals & Future Work

These are explicitly **not** part of v1. They may inform future versions but must not creep into v1 scope.

| Item | Rationale |
|---|---|
| `santa search` | Useful but not essential for the core declare-and-install workflow |
| Plugin system for community managers | v1 ships a fixed set of managers; extensibility comes later |
| Machine sync / dotfile management | Santa manages packages, not dotfiles |
| Lockfiles / version pinning | Package managers handle their own versioning |
| Windows as a supported platform | Best-effort only; macOS + Linux are the contract |
| Sickle 1.0 | Independent library, evolves on its own timeline |
| Sickle-CLI 1.0 | Developer utility, not part of the santa product |
| Dev-tools pipeline as public API | Internal tooling for maintaining package data |
| GUI / TUI interface | CLI only for v1 |
| Dependency resolution between packages | Santa delegates to underlying managers |

## Success Criteria

v1 is ready to ship when:

1. **All existing commands work reliably** — status, install, add, remove, config, completions, sources subcommands
2. **`santa init` exists** — generates a valid starter config on macOS and Linux
3. **Tier 1 managers pass end-to-end** — brew, apt, pacman, cargo, npm all generate correct scripts and report accurate status
4. **Tier 2 managers don't crash** — dnf, nix, aur, arch, scoop, winget, flathub may have incomplete features but must not panic or produce corrupt output
5. **Error messages are actionable** — config errors, missing managers, and network failures all produce clear guidance
6. **Config format is documented** — a new user can write a config from the docs alone
7. **Test coverage meets the bar** — per the test coverage table, no untested commands or Tier 1 managers
8. **Distribution works** — pre-built binaries, cargo install, and brew tap all install a working `santa` binary
9. **CI is green** — all checks pass on macOS and Linux
10. **No known panics or data loss** — safe-by-default execution model is intact, no path causes loss of user config
